# SLM Robustness Analysis on Orange Pi 5 Plus

This repository contains the material used to evaluate the robustness of **Small Language Model inference** on an **Orange Pi 5 Plus** board under different execution conditions.

The experiments compare two execution pipelines:

- **CPU pipeline**: GGUF models executed through `llama.cpp` / `llama-server`.
- **NPU pipeline**: RKLLM models executed through RKLLama and the Rockchip RKLLM runtime.

The repository includes raw CSV data, benchmark scripts, plotting scripts, generated plots, and references to the model archives used in the experiments.

## Setup

The experiments were executed on an **Orange Pi 5 Plus** board equipped with a **Rockchip RK3588** system-on-chip.

**Hardware setup:**

- **Orange Pi 5 Plus**
- **Rockchip RK3588**
- **ARM big.LITTLE CPU architecture**
- **Cortex-A76 high-performance cores**
- **Cortex-A55 efficiency cores**
- **Rockchip NPU**, up to 6 TOPS
- **16 GB RAM**

**Software setup:**

- Ubuntu Jammy-based Orange Pi image
- Python 3
- `stress-ng`
- `llama.cpp` / `llama-server` for CPU inference
- RKLLama and RKLLM runtime for NPU inference

Install the required system packages with:

```bash
sudo apt update
sudo apt install -y python3 python3-pip stress-ng curl
```

The scripts are intended to be executed on the Orange Pi board. Local paths inside the scripts may need to be adapted depending on the installation directory of the models, `llama.cpp`, and RKLLama.

## Workload

The workload consists of repeated text generation requests executed on **Qwen2.5 instruction models** with different parameter sizes:

- **Qwen2.5-0.5B**
- **Qwen2.5-1.5B**
- **Qwen2.5-3B**
- **Qwen2.5-7B**

The same benchmark workload is used across CPU and NPU experiments in order to compare the behavior of the two deployment pipelines under similar conditions.

The final experiments use three generation temperature values:

```text
0.0
0.5
1.0
```

For each final configuration, **30 repetitions** are collected.

The benchmark collects the following metrics in CSV format:

- **`ttft_s`**: Time To First Token.
- **`tgs_tokens_s`**: generation throughput in tokens per second.
- **`total_latency_s`**: total request latency.
- **`rss_peak_mb`**: peak resident memory usage.
- **`cpu_avg_percent`**: average host CPU usage.
- **`temp_avg_c`**: average board temperature.
- **`pass1`**: correctness indicator for the generated answer.
- **`ppl`**: perplexity value, where available.

For the NPU/RKLLM pipeline, perplexity is not computed directly by the RKLLM server, so the corresponding CSV field is left empty.

## Experimental conditions

Three execution conditions are evaluated:

1. **Baseline**
2. **Stress**
3. **CPU isolation / CPU pinning**

## Baseline

The baseline condition evaluates model inference without external stressors.

Baseline experiments include both **cold** and **warm** execution:

- **Cold execution**: the inference server is started before the measured run.
- **Warm execution**: an initial warm-up request is executed before the measured runs.

The cold/warm comparison is mainly used to evaluate the impact on **Time To First Token**.

**CPU baseline script:**

```text
script/cpu/run_benchmark_server_en.sh
```

**NPU baseline script:**

```text
script/npu/run_benchmark_npu_qwen.sh
```

## Stress

The stress condition evaluates model inference while a synthetic system workload is executed in parallel through `stress-ng`.

The final stressors are:

```text
cpu_w5
cache_w5
memcpy_w5
open_w5
itimer_w5
```

They correspond to:

- **`cpu_w5`**: CPU stress with 5 workers.
- **`cache_w5`**: cache stress with 5 workers.
- **`memcpy_w5`**: memory-copy stress with 5 workers.
- **`open_w5`**: file open/close stress with 5 workers.
- **`itimer_w5`**: interval timer stress with 5 workers.

The actual `stress-ng` commands used by the scripts are:

```bash
stress-ng --cpu 5 --metrics-brief
stress-ng --cache 5 --metrics-brief
stress-ng --memcpy 5 --metrics-brief
stress-ng --open 5 --metrics-brief
stress-ng --itimer 5 --metrics-brief
```

**CPU stress script:**

```text
script/cpu/run_stress_final.sh
```

**NPU stress script:**

```text
script/npu/run_stress_npu.sh
```

For the final stress experiments, these stressors are executed while the benchmark scripts collect inference metrics for all selected models and temperature values.

## CPU isolation / CPU pinning

The CPU isolation condition evaluates the effect of separating the inference process from the stress workload using CPU affinity.

In this repository, this condition is also referred to as **CPU pinning**.

The scripts use `taskset` to bind the inference process and the stress workload to different CPU groups:

- **Inference process**: CPU cores `4-7`.
- **Stress workload**: CPU cores `0-3`.

**CPU isolation scripts:**

```text
script/cpu/run_stress_cpu_pinning.sh
script/npu/run_stress_npu_pinning.sh
```

For the CPU pipeline, the `llama-server` process is launched on CPU cores `4-7`.

Concrete command structure used for CPU inference pinning:

```bash
taskset -c 4-7 ./llama-server \
  -m ../../models/cpu_gguf/qwen2.5-0.5b-instruct-q4_k_m.gguf \
  --host 127.0.0.1 \
  --port 8080
```

The stress workload is launched on CPU cores `0-3`:

```bash
taskset -c 0-3 stress-ng --cpu 5 --metrics-brief
```

For the NPU pipeline, the RKLLM/RKLLama benchmark process is launched with CPU affinity on cores `4-7`.

Concrete command structure used for NPU pinning:

```bash
taskset -c 4-7 python3 benchmark_server_npu.py \
  --model ../../models/npu_rkllm/qwen2.5-0.5b-instruct-rk3588-w8a8.rkllm \
  --temperature 0.0 \
  --mode warm \
  --runs 30
```

The stress workload is launched on CPU cores `0-3`:

```bash
taskset -c 0-3 stress-ng --cpu 5 --metrics-brief
```

For the final CPU isolation experiments, the scripts repeat this configuration across all selected models, temperatures, and stressors.

## How to reproduce the experiments

### 1. Clone the repository

```bash
git clone https://github.com/maraiaale/SLM-Robustness-OrangePi.git
cd SLM-Robustness-OrangePi
```

### 2. Install system dependencies

```bash
sudo apt update
sudo apt install -y python3 python3-pip stress-ng curl
```

### 3. Prepare the model files

Download the external model archives listed in:

```text
models/MODEL_LINKS.md
```

Then extract the CPU GGUF models into:

```text
models/cpu_gguf/
```

and the NPU RKLLM models into:

```text
models/npu_rkllm/
```

### 4. Run CPU baseline experiments

```bash
cd script/cpu
chmod +x run_benchmark_server_en.sh
./run_benchmark_server_en.sh
```

### 5. Run CPU stress experiments

```bash
cd script/cpu
chmod +x run_stress_final.sh
./run_stress_final.sh
```

### 6. Run CPU isolation experiments

```bash
cd script/cpu
chmod +x run_stress_cpu_pinning.sh
./run_stress_cpu_pinning.sh
```

### 7. Run NPU baseline experiments

```bash
cd script/npu
chmod +x run_benchmark_npu_qwen.sh
./run_benchmark_npu_qwen.sh
```

### 8. Run NPU stress experiments

```bash
cd script/npu
chmod +x run_stress_npu.sh
./run_stress_npu.sh
```

### 9. Run NPU isolation experiments

```bash
cd script/npu
chmod +x run_stress_npu_pinning.sh
./run_stress_npu_pinning.sh
```

## Notes

The CPU and NPU pipelines use different model formats because they target different runtime environments.

The CPU pipeline uses **GGUF models** executed through `llama.cpp`, while the NPU pipeline uses **RKLLM models** compatible with the Rockchip runtime.

For this reason, the results should be interpreted as a practical comparison between two deployment pipelines on the same embedded board, rather than as a strict comparison between identical binary model formats.
